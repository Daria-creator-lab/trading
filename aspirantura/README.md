# Функциональная архитектура с использованием ROS для автономных БПЛА

Йохвани Густав, Джами Чахал и Ассия Белбачир
Кафедра аэрокосмических систем, IPSA, Иври-сюр-Сен, Франция

Ключевые слова: Функциональная архитектура, БПЛА, Квадрокоптер, ROS.

Аннотация:

Беспилотные летательные аппараты (БПЛА) используются в нескольких приложениях благодаря своей стабильности и универсальности. В этой статье мы разработали функциональную архитектуру для автономных БПЛА с использованием Robot Operating System (ROS). Благодаря своей гибкости и простой в использовании реализации, наша архитектура упрощает внедрение автономного поведения для любого типа БПЛА. Эта иерархическая архитектура разделена на три уровня: уровень принятия решений, управления и восприятия. В этой статье все уровни и их реализация в ROS объяснены и детализированы.

## 1 ВВЕДЕНИЕ

В настоящее время применение беспилотных летательных аппаратов (БПЛА), также называемых дронами, охватывает不断扩大ющуюся сферу. Дроны особенно ценятся за их экономичность, стабильность, полу- или полную автономность и способность нести грузы. Действительно, имея лишь несколько двигателей, создающих подъемную силу в одном направлении, теория управления позволяет дронам выполнять специфические действия, такие как зависание, перемещение в заданную позицию, следование по траектории, вращение по кругу вокруг точки и т.д. Кроме того, дроны могут нести несколько датчиков, таких как лидар, монокамера, стереокамера или сонар, в зависимости от их миссии. Было разработано и испытано несколько миссий дронов, таких как поиск и спасение (Misra et al., 2020), мониторинг (Wang et al., 2019) или исследование (Maciel-Pearson et al., 2019).

В работе (Misra et al., 2020) авторы разработали роевое взаимодействие БПЛА для выполнения селективного исследования. Этот подход применяется при поиске и спасении для улучшения обнаружения выживших на земле. В работе (Wang et al., 2019) авторы предлагают использование БПЛА с камерой высокого разрешения для мониторинга океанской среды. Метод суперпикселей и сверточных нейронных сетей (CNN) используется для улучшения контроля распространения водорослей. В работе (Maciel-Pearson et al., 2019) авторы разработали автономный БПЛА для исследования внешней среды с помощью глубокого обучения с подкреплением. Этот подход использует Deep Q-Network для сокращения времени исследования.

БПЛА также используются для миссий, которые опасны и для мест, труднодоступных для человека. Для выполнения этих миссий БПЛА могут управляться дистанционно или быть полностью автономными. Автономным

БПЛА необходима надежная функциональная организация для обеспечения взаимодействия между восприятием (датчики), стабилизацией полета при достижении желаемого местоположения (управление) и принятием решений (в зависимости от стратегии). Организация опирается на прочную архитектуру, гарантирующую эффективность размещения всех программных компонентов и надежность системы. Таким образом, в литературе можно найти несколько типов функциональных архитектур, таких как субсумпционная архитектура (Brooks, 1986), иерархическая архитектура (Alami et al., 1998) и др. Более подробная информация о плюсах и минусах архитектур для БПЛА была объяснена в работе (Asmaa et al., 2019).

В этой статье мы решили разработать иерархическую архитектуру из-за ее надежности и простоты реализации с использованием Robot Operating System (ROS). Эта иерархическая архитектура позволяет нашему БПЛА организовывать команды на трех уровнях. Каждый уровень имеет свою роль и свою важность. Чем выше уровень, тем выше его цель. Высокий уровень (принятие решений) имеет информацию, относящуюся ко всей миссии, поэтому он имеет большее время рассуждения. Однако нижний уровень имеет сокращенное время для рассуждений, но более реактивен в краткосрочной миссии, такой как избегание препятствий. Цель этого фреймворка — быть гибким для любого типа БПЛА и миссий. Мы полагаемся на симулятор Gazebo для проверки предложенной архитектуры.

ROS — это открытая мета-операционная система, предоставляющая экосистему и набор инструментов для робототехнических приложений. ROS обеспечивает связь между программами, называемыми узлами. Эта связь централизована вокруг одного узла, называемого мастером. Каждый

Благодаря своей структуре реализация ROS является надежной, расширяемой и выигрывает от инструментов графического анализа.

Разработка функциональной архитектуры в стабильном фреймворке, таком как ROS, обеспечивает получение надежной системы. Эта ассоциация была выполнена с субсумпционной архитектурой в (Li et al., 2016; Yi et al., 2016) и с архитектурой T-Rex (Mcgann et al., 2008). В этой статье мы представляем нашу разработанную архитектуру, которая разделена на три уровня: восприятие (нижний уровень), управление (средний уровень) и уровень принятия решений (верхний уровень). Эта архитектура реализована для случая использования локализации очага возгорания.

Валидация подхода уже была выполнена в (Belbachir and Escareno, 2016), однако архитектура не была реализована на БПЛА. Таким образом, в этой статье мы предлагаем расширить предыдущую стратегию исследования в нашу функциональную архитектуру на основе ROS.

Эта статья организована следующим образом. Раздел II объясняет проектирование фреймворка для приложений БПЛА. Раздел III детализирует, как этот фреймворк реализован. Раздел IV представляет выводы и будущую работу.

### 3.1 Robot Operating System

Robot Operating System (ROS) — это среда с открытым исходным кодом, которая позволяет нам создавать сложное и надежное поведение роботов, используя набор библиотек и инструментов. Таким образом, основная программа может быть разделена на несколько подпроцессов, которые будут выполняться параллельно и взаимодействовать друг с другом.

Узлы. Узлы представляют процессы. Каждый из них имеет определенную задачу для повышения эффективности всей системы. Связь между этими узлами осуществляется с помощью набора топиков и сервисов.

Топики и Сервисы. Топики — это мосты, через которые информация, называемая сообщениями, отправляется от одного узла к другому. Узлы публикуют и/или подписываются на топики, чтобы соответственно отправлять и/или получать эти сообщения. Топик может принимать только сообщения, тип которых указан пользователем. Например, узел, публикующий строки, не может публиковать целые числа в том же топике.

Сервисы основаны на модели запрос/ответ: узел предоставляет сервис, который будет вызван другим узлом с использованием имени сервиса. Узел, создавший сервис, называется сервером, а узлы, вызывающие сервис, называются клиентами. Когда сервер получает запросы от клиента, он анализирует их и вычисляет конкретную задачу, чтобы отправить ответ клиенту.

Топики следует использовать для непрерывных потоков данных, тогда как сервисы являются блокирующими вызовами, и поэтому их следует использовать для вызовов удаленных процедур, которые быстро завершаются.

### 3.2 Gazebo

Gazebo 3 — это симулятор, предлагающий возможность проектировать роботов в помещениях/на открытом воздухе, которые могут быть либо загружены из репозиториев баз данных, либо разработаны пользователями. Gazebo может генерировать данные датчиков, такие как сонар, лазерный дальномер или камера, и к каждому датчику может быть добавлен шум. Кроме того, могут быть добавлены физические силы, такие как ветер, особенно для летательных аппаратов. Таким образом, робот находится в условиях, близких к реальности, чтобы проверить надежность разработанных программ.

### 3.3 ArduPilot

Разработанный контроллер не был реализован на ROS. Мы решили использовать надежное существующее решение, предоставляемое ArduPilot 4, открытой системой автопилота, которая предоставляет контроллеры для широкого спектра транспортных средств, таких как дроны, подводные и наземные транспортные средства, самолеты, лодки и вертолеты.

Их контроллер хорошо подходит для управления БПЛА в любом типе среды (в помещении/на открытом воздухе). Большое

количество систем мульти-БПЛА основаны на этом контроллере (R. Braga, 2017) (Sardinha et al., 2018). Более того, их контроллер точно реализует тот, который мы подробно описали в предыдущем разделе. Для использования ArduPilot (например, взлета) необходимо отправить определенные команды в ArduPilot. Одна из них — включить двигатели, а другая — взлететь дрону. То же самое применимо, если пользователь хочет посадить БПЛА. ArduPilot предоставляет БПЛА несколько режимов полета. Основные режимы описаны ниже:

## СТАБИЛИЗАЦИЯ (STABILIZED)

БПЛА полностью управляется пользователем, что означает, что непрерывные команды должны отправляться контроллеру, чтобы БПЛА летел. В противном случае он останется на земле или, что еще хуже, разобьется. Таким образом, этот режим в основном выбирается, если пользователь хочет управлять БПЛА вручную.

## УДЕРЖАНИЕ ВЫСОТЫ (ALT_HOLD)

Это низкоуровневый автономный режим: он обеспечивает удержание БПЛА своей высоты. Однако могут происходить движения вперед/назад и влево/вправо.

## УДЕРЖАНИЕ МЕСТОПОЛОЖЕНИЯ (LOITER)

Аналогично, БПЛА удерживает свою высоту, но также и свое положение. Например, если БПЛА необходимо поддерживать свое положение во время выполнения обработки изображений с использованием данных от встроенной камеры, этот режим может быть вариантом.

## АВТОМАТИЧЕСКИЙ (AUTO)

БПЛА достигает заранее заданного набора путевых точек. Например, этот режим может быть активирован, когда задача связана с мониторингом.

## УПРАВЛЯЕМЫЙ (GUIDED)

БПЛА автономно перемещается из своего текущего положения в полученную желаемую позу (положение + ориентация). Определяя стратегию исследования, следующая позиция, которую должен достичь БПЛА, может отправляться на каждой итерации.

## ПОСАДКА (LAND)

БПЛА уменьшает свою высоту, пока не достигнет земли, вертикально. Затем двигатели автоматически выключаются.

Для нашей архитектуры мы использовали режим GUIDED, в котором команды отправляются нашим уровнем принятия решений. Все команды, перечисленные выше, отправляются контроллеру через протокол связи, называемый Micro Areal Vehicle Link (MAVLink) (Lamping et al.,

2018). Поэтому мы использовали этот протокол для настройки связи между нашими узлами и этим контроллером.

### 3.4 Интеграция архитектуры в ROS

## Узлы

Мы решили протестировать нашу архитектуру в смоделированной среде. Мы работали с моделью квадрокоптера, интегрированной в Gazebo.

То, как организован ROS, привело к разделению нашей системы на четыре узла следующим образом:

1. Стратегия (Strategy). Он иллюстрирует поведение модуля Exploration Strategy, представленного на Рисунке 1. Strategy получает текущее положение и ориентацию квадрокоптера в его смоделированной среде \([x_{m e s},y_{m e s},z_{m e s},\theta_{m e s},\theta_{m e s},\psi_{m e s}]^{l}\) путем подписки на топик /mavros/local_position/pose. Эта поза выражается относительно системы отсчета симулятора. Если квадрокоптер имеет бортовые датчики, он может затем анализировать среду, в которой он работает, и действовать соответственно. Данные, исходящие от датчиков, извлекаются путем подписки на топики, генерируемые симулятором. В зависимости от реализованной стратегии этот узел определяет конфигурацию \([x_{d},y_{d},z_{d},\psi_{d}]^{l}\), которую должен иметь квадрокоптер, как объяснено в разделе 2. Затем он публикует желаемую позу для контроллера в топик /mavros/setpoint_position/local.

Как объяснено в разделе 3.3, контроллеру необходимо получать команды, чтобы поднять квадрокоптер в воздух, управлять им и затем посадить. Таким образом, Strategy вызывает сервис /mavros/cmd/arming для включения двигателей, /mavros/cmd/takeoff для поднятия БПЛА в воздух и /mavros/cmd/land для его посадки.

2. mavros. Это мост между ROS и контроллером. Этот узел использует MAVLink для настройки

связи между узлами, выполняемыми на ROS, и контроллером квадрокоптера, как объяснено в разделе 3.3.

mavros подписывается на топик /mavros/local_position/pose, чтобы получить желаемую позу квадрокоптера. Он публикует текущую позу квадрокоптера в топик /mavros/local_position/pose, а также информацию о его батарее в топик /mavros/battery.

Кроме того, он также работает как сервер и ожидает, пока клиент отправит запрос на несколько сервисов: /mavros/cmd/arming, /mavros/cmd/takeoff и /mavros/cmd/land. Эти сервисы вызываются Strategy, как обсуждалось ранее.

3. sim_vehicle.py. Это программное обеспечение в цикле запускает контроллер, разработанный ArduPilot, в смоделированной среде. Оно получает сообщения MAVLink от mavros и вычисляет отправленные команды для двигателей квадрокоптера.

Эти команды могут быть: включение двигателей, взлет, посадка, переход из текущей позы квадрокоптера в желаемую или достижение квадрокоптером набора путевых точек.

Большинство команд, полученных контроллером, зависят от встроенной стратегии. Каждая команда, отправленная на двигатели, содержится в сообщении MAVLink и затем интерпретируется в Gazebo. Аналогично, контроллер получает сообщения MAVLink от симулятора, чтобы получить позу квадрокоптера.

4. gazebo_ros. Этот узел запускает симулятор Gazebo, в котором наш квадрокоптер работает в нашей среде. Как указано выше, Gazebo получает команды MAVLink от контроллера для перемещения квадрокоптера соответствующим образом. Кроме того, он отправляет позу робота и состояние его батареи контроллеру.

В зависимости от датчиков, установленных на квадрокоптере, gazebo_ros будет публиковать данные датчиков в разных топиках.

## Топики

Как обсуждалось ранее, в топике может быть опубликован только один тип сообщения. Таким образом, топики и связанные с ними сообщения, которые мы использовали для связи между узлами, описаны ниже:

/mavros/septoint_position/local. Этот топик получает сообщения типа geometry_msgs/PoseStamped, параметрами которого являются координаты [x, y, z] в метрах и ориентация, выраженная в кватернионах. Кроме того, время, в которое было отправлено сообщение, а также система отсчета, в которой выражена поза, указываются узлом, публикующим сообщение.

/mavros/local_position/pose. Аналогично, этот топик получает сообщения типа geometry_msgs/PoseStamped.

## Сервисы

Следующие сервисы выполняются mavros (сервер).

/mavros/cmd/arming. Клиент использует тип сообщения mavros_msgs/CommandBool: в качестве запроса клиент должен указать true или false желаемое состояние двигателей. Либо они будут включены (true), либо выключены (false).

Сервер вернет клиенту логическое значение, указывающее, был ли обработан запрос, и целое число, которое определяет, был ли запрос успешно выполнен: 0, если успешно, и 4 в противном случае.

/mavros/cmd/takeoff. Клиент использует тип сообщения mavros_msgs/CommandTOL: в качестве запроса клиент должен указать желаемую высоту для квадрокоптера после взлета и может дополнительно определить желаемые координаты (широту, долготу, рысканье). Фактически, если координаты не указаны (все значения равны 0), квадрокоптер просто взлетит вертикально и стабилизируется на желаемой высоте.

/mavros/cmd/land. Аналогично, клиент использует тип сообщения mavros_msgs/CommandTOL: пользователь может присвоить 0 всем запрашиваемым параметрам, чтобы квадрокоптер приземлился вертикально. В противном случае он может указать желаемые координаты для посадки. Параметр высоты, определенный в запросе клиентом, не будет учитываться сервером, так как квадрокоптер приземляется.
